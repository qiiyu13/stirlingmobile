## 1. Core Entities

### 1.1 FileInfo

Represents a user file tracked by the app.

```kotlin
data class FileInfo(
    val id: String,              // UUID
    val displayName: String,     // "document.pdf"
    val uri: Uri,                // Content URI (SAF)
    val internalPath: String?,   // Copied-to path in app storage
    val sizeBytes: Long,
    val pageCount: Int?,
    val mimeType: String,
    val addedAt: Instant,
    val source: FileSource,      // See below
    val thumbnailPath: String?   // Cached thumbnail file path
)

enum class FileSource {
    SAF,            // User picked via system file picker
    TOOL_OUTPUT,    // Generated by a tool operation
    PIPELINE_STEP,  // Intermediate step in a pipeline
    RECENT          // From recent files list
}
```

### 1.2 ToolOperation

Records a completed tool run for history and undo.

```kotlin
data class ToolOperation(
    val id: String,              // UUID
    val toolId: String,          // "merge", "split", "rotate", etc.
    val inputFileIds: List<String>,
    val outputFileId: String?,
    val params: String,          // JSON of tool parameters
    val status: OperationStatus,
    val startedAt: Instant,
    val completedAt: Instant?,
    val errorMessage: String?
)

enum class OperationStatus { PENDING, PROCESSING, COMPLETED, FAILED, CANCELLED }
```

### 1.3 Pipeline

A chain of tool operations on the same document.

```kotlin
data class Pipeline(
    val id: String,
    val name: String,            // Auto-generated or user-named
    val steps: List<PipelineStep>,
    val createdAt: Instant,
    val currentStepIndex: Int
)

data class PipelineStep(
    val operation: ToolOperation,
    val inputFile: FileInfo,
    val outputFile: FileInfo?,
    val canUndo: Boolean
)
```

## 2. File Lifecycle

```
User picks file (SAF)
  │
  ├── Copy to /files/imported/{uuid}.pdf  (originals never modified)
  │
  ├── User opens tool (e.g., Rotate)
  │     │
  │     ├── Tool reads from /files/imported/{uuid}.pdf
  │     └── Tool writes to /files/working/{session_id}/{step}_output.pdf
  │
  ├── User chains another tool (e.g., Compress)
  │     │
  │     ├── Tool reads previous output
  │     └── Tool writes to /files/working/{session_id}/{step}_output.pdf
  │
  ├── User exports final
  │     ├── Copy to user-chosen SAF destination
  │     └── Share via Android share sheet (temporary content URI)
  │
  └── App closes / session ends
        └── /files/working/ contents deleted
        └── /files/imported/ retains orphan files for 7 days, then pruned
```

## 3. Temp File Policy

| Directory | Created | Deleted | Max Age |
|---|---|---|---|
| `/files/imported/` | On file import | Manual + auto-prune | 7 days unused |
| `/files/working/` | On tool start | On session end | Duration of session |
| `/files/temp/` | On demand (Rust engine) | On engine shutdown | Duration of engine lifecycle |
| `/cache/thumbnails/` | On first render | LRU eviction + clear cache | Unlimited (LRU) |
| `/files/ocr/tessdata/` | On language download | Manual delete | Permanent |

### 4.1 Tables

```sql
CREATE TABLE recent_files (

    id TEXT PRIMARY KEY,

    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

### 4.2 Queries

```kotlin
@Dao
interface FileDao {
    @Query("SELECT * FROM recent_files ORDER BY last_accessed_at DESC LIMIT 20")

}
```

## 5. Rust Engine State

The Rust engine is stateful within a session. It holds:

```rust
pub struct EngineState {
    temp_dir: PathBuf,
    open_documents: RwLock<HashMap<DocumentId, PdfDocument>>,
    operation_queue: Vec<Operation>,
    thread_pool: rayon::ThreadPool,
    cancel_flags: HashMap<OperationId, Arc<AtomicBool>>,
}
```

`PdfDocument` is an open, parsed PDF:

```rust
pub struct PdfDocument {
    pub path: PathBuf,
    pub doc: lopdf::Document,
    pub page_count: usize,
    pub metadata: DocumentMetadata,
    pub password: Option<String>,
}
```

Documents are loaded on first operation and cached until `engine_destroy()` or the session ends. The cache is limited to 5 open documents (LRU eviction of oldest on 6th open).

## 6. Undo Model

Undo is file-level, not operation-level. Each tool step writes to a new file. To "undo":

1. Pipeline tracks each step's input and output files
2. Undo means reverting the current file to the previous step's output
3. This is a file swap — no operation reversal needed

```kotlin
fun undo(pipeline: Pipeline): Pipeline {
    if (pipeline.currentStepIndex == 0) return pipeline // nothing to undo
    val previousStep = pipeline.steps[pipeline.currentStepIndex - 1]
    // Current working file becomes previous step's output
    // Current step marked as undone
    return pipeline.copy(currentStepIndex = pipeline.currentStepIndex - 1)
}
```

This is simple but disk-heavy for long pipelines. For v1 this is acceptable — pipelines rarely exceed 3-4 steps.

## 7. Serialization

- Tool parameters: JSON (`serde_json` in Rust, `kotlinx.serialization` in Kotlin)
- File metadata: JSON (shared schema between Rust and Kotlin)
- Room entities: SQLite (Room auto-mapping)
- Preferences: DataStore (protobuf / preferences key-value)
- Pipeline state: JSON (file-based, persisted in `/files/working/{session_id}/pipeline.json`)
