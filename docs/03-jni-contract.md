# Stirling Mobile — JNI Contract

> **Status:** Draft v1.0
> **Principle:** JNI surface is the API. Kotlin never touches raw PDF bytes — Rust owns all PDF state. All JNI calls are thread-safe, reentrant, and panic-safe.

---

## 1. Design Rules

1. **Rust owns the PDF** — All PDF objects live on the Rust heap. Kotlin only holds opaque handles (`Long` representing a raw pointer).
2. **No byte arrays across JNI for large files** — >10MB files pass file paths, not buffers.
3. **Every JNI call returns `EngineResult`** — never throws. Errors are typed, not String-based.
4. **Thread safety** — All engine operations are `&self` (immutable borrow) unless they mutate internal maps. The engine uses `RwLock` internally.
5. **Panic boundary** — Every `#[no_mangle]` function has a `std::panic::catch_unwind` wrapper that converts panics to error results.
6. **Memory ownership** — When a JNI call returns a byte array, Kotlin owns the buffer. When Kotlin passes a byte array, it is copied into Rust memory.

## 2. Common Types

```rust
// The result type returned by all engine operations
#[derive(Serialize)]
pub enum EngineResult {
    Ok { data: Vec<u8> },
    OkPath { path: String },
    OkMultiple { paths: Vec<String> },
    OkInfo { json: String },
    Err { code: ErrorCode, message: String },
}

#[derive(Serialize)]
pub enum ErrorCode {
    InvalidInput,
    CorruptedPdf,
    UnsupportedFeature,
    PasswordRequired,
    WrongPassword,
    MemoryLimit,
    IoError,
    InternalError,
}
```

```kotlin
// Kotlin side — manually defined for clarity, matches Rust exactly
sealed class EngineResult {
    data class Ok(val data: ByteArray) : EngineResult()
    data class OkPath(val path: String) : EngineResult()
    data class OkMultiple(val paths: List<String>) : EngineResult()
    data class OkInfo(val json: String) : EngineResult()
    data class Err(val code: ErrorCode, val message: String) : EngineResult()
}

enum class ErrorCode {
    InvalidInput, CorruptedPdf, UnsupportedFeature,
    PasswordRequired, WrongPassword, MemoryLimit,
    IoError, InternalError
}
```

## 6. Error Handling Convention

**Rust side**: Every public function returns `EngineResult`. Panics are caught.

**Kotlin side**: JNI calls wrapped in `runCatching`:

```kotlin
suspend fun merge(paths: List<String>, output: String): EngineResult =
    withContext(Dispatchers.Default) {
        runCatching {
            nativeMerge(paths.toTypedArray(), output)
        }.getOrElse { e ->
            EngineResult.Err(ErrorCode.InternalError, e.message ?: "Unknown JNI error")
        }
    }
```

**ViewModels** handle `EngineResult.Err` by mapping to user-facing error states. Never show raw error codes — use i18n'd messages.

## 7. Memory Model

| Data size | Transfer method | Where data lives |
|---|---|---|
| < 10MB | ByteArray via JNI | Copied. Kotlin → Rust heap (input), Rust → Kotlin heap (output) |
| > 10MB | File path string | Written to disk by Rust, read by Kotlin. No copy. |
| Temp files | Engine writes, Kotlin reads | `/data/data/com.stirlingmobile/files/temp/` |
| Opaque handles | `Long` (pointer cast) | Rust heap, freed by explicit `destroy` call or engine shutdown |

## 8. Threading Guarantees

| Guarantee | Implementation |
|---|---|
| All engine functions are thread-safe | `RwLock` on internal state maps |
| No JNI call blocks indefinitely | 30-second timeout enforced in Kotlin coroutine wrapper |
| Concurrent operations allowed | v1: one PDF operation at a time, serialized in Rust; further calls queue in Kotlin. v1.1+: revisit with rayon thread pool if benchmarks show it's needed (see R9, 14-risk-register.md) |
| UI never blocked | All JNI calls on `Dispatchers.Default`, results posted to Main |
| PDF rendering is cancelable | Kotlin coroutine cancellation → Rust checks `AtomicBool` flag between pages |
